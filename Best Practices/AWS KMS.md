# **AWS KMS (Key Management Service)**  

## **Overview**  
AWS Key Management Service (**KMS**) is a fully managed encryption service that helps you create, manage, and control cryptographic keys across AWS services and applications.

---

## **Key Concepts**  

| **Term** | **Description** |
|----------|---------------|
| **Customer Master Key (CMK)** | The primary key entity in AWS KMS, used to generate and manage encryption keys. |
| **Symmetric CMK** | Uses a **single key** for encryption and decryption. |
| **Asymmetric CMK** | Uses a **public-private key pair** for encryption, decryption, signing, and verification. |
| **AWS Managed Keys** | CMKs managed automatically by AWS for specific services (e.g., S3, RDS). |
| **Customer Managed Keys** | CMKs fully controlled by the user, including policies, rotation, and lifecycle management. |
| **Data Keys** | Keys generated by CMKs used for encrypting data, ensuring that the CMK itself is not exposed. |
| **Envelope Encryption** | The process of encrypting data with a data key, which is then encrypted using a CMK. |
| **Key Policies** | IAM-like policies that define permissions on KMS keys. |

---

## **Key Types in AWS KMS**  

| **Key Type** | **Use Case** |
|-------------|-------------|
| **AWS Managed CMK** | Default encryption keys for AWS services like S3, RDS, and Lambda. |
| **Customer Managed CMK** | User-controlled keys for custom encryption needs. |
| **Symmetric CMK** | Encrypts and decrypts data with a single key (AES-256). |
| **Asymmetric CMK** | Supports **RSA and ECC** for digital signatures and public-key encryption. |
| **HMAC CMK** | Used for cryptographic message authentication with hashing. |

---

## **Common AWS KMS Use Cases**  

✅ **Encrypting Data at Rest** – Secure S3 objects, RDS databases, EBS volumes, and DynamoDB tables.  
✅ **Encrypting Data in Transit** – Use TLS certificates and secure API communications.  
✅ **Envelope Encryption** – Protect sensitive data using key hierarchies.  
✅ **Digital Signatures** – Verify integrity and authenticity of documents and transactions.  
✅ **Secure Application Secrets** – Store and retrieve API keys, credentials, and sensitive configurations.

## **Integrations with AWS Services**  

AWS KMS integrates with multiple AWS services for seamless encryption:  

| **Service** | **Integration Type** |
|------------|--------------------|
| **Amazon S3** | Server-side encryption (SSE-KMS) |
| **Amazon RDS** | Transparent data encryption (TDE) |
| **Amazon EBS** | Volume encryption |
| **AWS Lambda** | Environment variable encryption |
| **DynamoDB** | Table encryption |
| **CloudTrail** | Log integrity validation |
| **Secrets Manager** | Secure storage for secrets and credentials |

---

## **Key Rotation & Security Best Practices**  

🔹 **Enable Key Rotation** – AWS Managed CMKs rotate automatically every 1 year; Customer Managed CMKs can be set for manual rotation.  
🔹 **Use Least Privilege Access** – Restrict key access with KMS policies and IAM roles.  
🔹 **Monitor Key Usage** – Enable AWS CloudTrail to log KMS API calls.  
🔹 **Use Grants for Temporary Access** – Instead of updating key policies, use grants for limited, scoped permissions.  
🔹 **Limit Cross-Account Access** – Use resource-based policies for controlled sharing.  

---

## **KMS API Cheat Sheet**  

| **API Action** | **Description** |
|---------------|---------------|
| `Encrypt` | Encrypts plaintext data using a KMS key. |
| `Decrypt` | Decrypts ciphertext using a KMS key. |
| `GenerateDataKey` | Generates a data key for envelope encryption. |
| `ReEncrypt` | Re-encrypts data from one KMS key to another. |
| `CreateKey` | Creates a new customer-managed CMK. |
| `DescribeKey` | Retrieves metadata about a KMS key. |
| `ListKeys` | Lists all available KMS keys. |
| `ScheduleKeyDeletion` | Schedules the deletion of a KMS key (pending for 7–30 days). |

---

## **Choosing the Right Key Management Approach**  

| **Scenario** | **Recommended Approach** |
|-------------|--------------------------|
| Encrypting AWS services data | AWS Managed CMKs |
| Full control over keys | Customer Managed CMKs |
| Public-key cryptography | Asymmetric CMKs |
| Short-term encryption keys | GenerateDataKey API |
| HMAC authentication | HMAC CMKs |

# Attribute-Based Access Control for XRAY Image Decryption

You have just used **attribute-based access control (ABAC)** to grant medical technicians the ability to decrypt XRAY images while working on their **EC2 instance**.

## IAM Policy

Below is the IAM policy that enforces this access control:

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Action": [
                "s3:read*",
                "s3:list*",
                "s3:describe*",
                "s3:getobject*",
                "s3:putobject*",
                "kms:CreateKey",
                "kms:Decrypt",
                "kms:Encrypt",
                "kms:GenerateDataKey",
                "kms:DescribeKey",
                "dynamodb:DescribeTable",
                "dynamodb:CreateTable",
                "dynamodb:ListTables",
                "dynamodb:PutItem",
                "dynamodb:Query",
                "dynamodb:Scan"
            ],
            "Resource": "*",
            "Effect": "Allow"
        },
        {
            "Condition": {
                "StringNotEqualsIgnoreCase": {
                    "aws:ResourceTag/ImageType": "XRAY"
                }
            },
            "Action": [
                "kms:Decrypt"
            ],
            "Resource": "arn:aws:kms:us-east-1:247769687189:key/*",
            "Effect": "Deny"
        }
    ]
}
```

## Explanation

- The **first statement** allows broad permissions to access **S3**, **KMS**, and **DynamoDB** for necessary operations.
- The **second statement** enforces a **deny condition** on `kms:Decrypt` if the `ImageType` tag does **not** match `"XRAY"`, ensuring that **only XRAY images can be decrypted**.

This approach **ensures security** by leveraging **resource tags** for access control, following AWS **best practices** for fine-grained permissions.

## **Final Tips**  

✔ Use **CloudTrail logs** to audit KMS API actions.  
✔ Store **sensitive secrets** in AWS Secrets Manager, **not in KMS key policies**.  
✔ Set **IAM least privilege** policies for key access control.  
✔ Avoid deleting CMKs immediately—schedule deletion with a **waiting period**.  
✔ Encrypt **sensitive application data** before storing it in databases or file storage.
